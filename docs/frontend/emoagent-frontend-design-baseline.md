# EmoAgent 前端设计基准文档

> **用途**：前端开发的设计总纲与唯一基准。任何重新开发（无论 CC 还是人工）都以此为准，**不再自由发挥配色与布局**。
> **核心原则**：本文档不只规定「做成什么样」，更规定「为什么这样」——理由写清楚，跑偏的空间才小。每条规则后附依据。
> **配套**：实现工程规格见 `frontend-cc-spec.md`；接口契约见其 §3。

---

## 0. 一句话定位

EmoAgent 有两个物理分离的前端：**学生端**（面向初中生的情感陪伴产品，温暖、看不到机制）与**研究分析台**（面向研究者/开发者/评委的离线工具，机制全开）。二者共享一个后端 `/chat`，中间隔一道刻意设计的**安全/隐私之墙**。

> **为什么分离而非合并**：学生端的目标是「让孩子忘记背后有系统」，分析台的目标是「让人看清系统」——一个要藏、一个要露，天然对立。物理分离让产品的温暖与研究的严谨各自纯粹，也守住了「未成年人数据保护」的红线（孩子用的入口永远接触不到打分与危机信号）。这道墙本身是论文里安全设计的一个论据。

## 1. 主题设计（视觉系统 · 已锁定）

### 1.1 设计哲学：简约而不简单

提炼自对 Claude / ChatGPT / Gemini 三家的研究：

- **Claude**：暖中性打底（非纯白）+ 一种有性格的颜色仅作点睛 + 字体有态度 → 温暖而不冷。
- **ChatGPT**：极致克制，平静来自间距/圆角/节奏的精确，不靠加颜色。
- **Gemini（Neural Expressive）**：把 AI 回复当作「被设计的对象」，而非聊天记录的一行 → 层级优先于均质。

> **共识（也是本系统的设计铁律）**：① 几乎不用纯黑纯白，用带暖/冷的中性；② 主色极度克制，大面积是中性，颜色只点睛；③ 质感来自字体+间距+圆角的精确，不来自堆砌；④ 层级是设计核心。

### 1.2 配色（鼠尾草绿点睛体系）

| 角色 | 色值 | 用途 | 为什么 |
|---|---|---|---|
| 暖白底 | `#faf8f3` | 占画面 ~90% 的主背景 | 暖中性、非纯白，温暖不浑浊（Claude 式）。大面积干净中性是「不发霉、不廉价」的前提 |
| 暖白替代 | `#f3f0e8` | 渐变下沿、侧栏底 | 同上，制造极轻微的纵深 |
| 卡片白 | `#fffefb` | 卡片、输入框 | 最暖的白，非 `#fff` |
| 软墨 | `#33312c` | 正文文字 | 近炭非纯黑，降低对比的攻击性 |
| 次级墨 | `#6a675f` | 辅助文字 | — |
| 鼠尾草绿 | `#6f9c80` | **仅点睛**：logo点、AI标记点、发送键、一处氛围光 | 绿在色彩心理学中降焦虑；去饱和版清新而不刺激 |
| 深鼠尾草 | `#4a6e58` | AI 标记文字、强调 | — |
| 雾霾蓝 | `#8fb0bf` | 第二点缀（如情境色） | 低饱和多 pastel 协同，提供「清新」而不靠单色 |
| 柔陶土 | `#c08a6a` | 第三点缀 | 暖意，小剂量 |
| 转介赤陶 | `#bd7a5e` | 危机面板 | 暖而非警报红，分量来自排版非饱和度 |

> **配色铁律（这是「发霉」教训的固化）**：鼠尾草绿**绝不大面积填充**（不做聊天气泡主色）。曾经把去饱和绿铺成气泡主色 + 配偏黄底 → 黄+灰绿=霉斑感。正解：大面积干净暖白 + 绿只出现在几个点。**改色前必读此条。**
>
> **色彩心理学依据**：高饱和暖色（橙/黄/火红）会升高生理唤起、加重焦虑波动，疗愈空间应避开；纯冷蓝灰则显沉闷（Headspace 明确反对「沉闷的蓝灰海洋」）。真正的解 = **低饱和多 pastel + 暖中性打底**——清新来自色彩多样，放松来自全部去饱和，二者不冲突。

### 1.3 工艺规则（Headspace 式，不可违反）

- 不用纯黑、不用纯白。
- **阴影带色不带灰**（如气泡阴影用 sage 色调），灰阴影显廉价。
- 圆形语言：圆角统一、无尖角。
- 每个颜色服务单一情绪目标，不无意义铺色。

### 1.4 字体与排版

- 中文正文：`Nunito` + `PingFang SC` / `Noto Sans SC`，圆润无衬线，亲和。
- **AI 回复是「被设计的对象」**：字号 17px、行距 1.85、带 `EmoAgent` 小标记、左对齐独立成段。
- 用户消息：右侧低调浅灰气泡（`#eef0ea`），字号 15px，刻意比 AI 回复轻。

> **为什么 AI 回复要更重**（Gemini 课）：让「孩子说 → AI 郑重回应」有仪式感。若两者同等重量，就退回成冷冰冰的聊天记录，失去「被认真对待」的体验。这是层级，不是装饰。

## 2. 学生端 · 关键功能

### 2.1 功能清单与理由

| 功能 | 内容 | 为什么要有 |
|---|---|---|
| **对话主区** | 只渲染 `reply_text`；AI 回复大字独立成段 | 产品的核心；层级体现「被认真对待」 |
| **侧边栏 + 历史会话** | 列本设备最近会话，标题取**首条真实倾诉**；「开启新对话」 | 填充画面（解决「空」），帮助回到刚才的话题；不暗示跨会话画像 |
| **低门槛起手式** | 空状态给「今天有点累 / 想吐槽一件事 / 只是想有人在」等入口 | 初中生面对空输入框常不知从何说起；降低开口门槛（Headspace「让开始不可怕」） |
| **我聊过的** | 只展示本设备本地会话列表，可「让我忘记」清除 | 会话历史，不做情绪分析、画像归因或长期记忆，避免诱导依赖和固化标签 |
| **呼吸小工具** | 纯前端 8s 一吸一呼动画 | 纯陪伴；对应 CASEL「自我管理」；无后端依赖 |
| **危机转介面板** | yellow/red 时替换输入框，见 §2.3 | 安全红线落地，论文「未成年人安全门设计」的可见证据 |

> **空状态设计（采纳截图式修订）**：顶部「你可以慢慢说，我会认真听」+ 主区温和开场白 + 底部起手式 pill。呼吸只作为独立工具出现，不在启动界面重复出现。

### 2.2 会话上下文与长期记忆边界

| 范围 | 内容 | 为什么这样设计 |
|---|---|---|
| **会话内短期上下文（保留）** | 同一 `session_id` 多轮复用；后端 Redis 窗口（`HISTORY_WINDOW_N=6`，TTL 默认 7 天）供 F1-F4 看最近上下文 | F1 需要跨轮累积危机信号；这是短期上下文，不是长期记忆 |
| **本地会话历史（保留）** | 学生端 `localStorage` 仅保存本设备会话标题与消息文本，可被「让我忘记」清除 | 方便回到刚才的话题；不包含后端分析字段，不上传画像 |
| **跨会话画像/长期记忆（不做）** | 不持久化“这个孩子是什么类型”、高频情境、情绪轨迹或个性化画像 | 避免诱导情感依赖、避免固化波动期青少年标签、最小化未成年人敏感数据留存 |

> **论文/答辩表述**：系统刻意不持久化用户画像，以避免诱导情感依赖、避免对波动期青少年施加固化标签，并最小化未成年人敏感数据留存。面向孩子隐藏评估机制，把透明性留给监管者 / 研究者（console），是双前端物理隔离的理论依据。

### 2.3 危机转介（固定不可改）

- 触发：`risk_level` 为 yellow/red。
- 行为：渲染 `reply_text` 后弹出转介面板，**替换并锁定输入框**。
- 内容：标题 + 共情句 + 可执行引导（联系可信成年人）+ 两个可拨号按钮 `tel:12356` / `tel:12355`；red 额外提示 120/110。
- 号码**硬编码**，文案固定，不经任何动态生成。

> **为什么锁输入、不展开**：呼应 C-SSRS 原则「红色绝不自行干预，AI 不是危机干预者，只转介」。为什么暖色不警报红：呼应「分级而非一刀切，不过度反应」。为什么必含可信成年人：呼应「不在孩子与可信成年人间制造隔离」。

### 2.4 学生端绝对禁止（安全铁律）

- **类型层只允许** `reply_text` / `risk_level` / `session_id` 三字段；`scores`/`candidates`/`weighted_total` 等分析字段在类型与渲染双重丢弃。
- **不得有任何通往机制/分析/分数的入口**（按钮、图标、链接均不可）。
- 失败兜底只显示安全话术，**绝不暴露原始错误或 `failure_reason`**。

> 这是儿童安全约束，优先级高于一切视觉与功能诉求。

## 3. 研究分析台 · 关键功能

### 3.1 三视图与双重用途

| 视图 | 内容 | 对评委 | 对开发者 |
|---|---|---|---|
| **单轮追踪** | 输入 → F1分级 → F2情境+激活CASEL → F3双候选 → F4 EPITOME打分·择优 → 输出（首尾闭环） | 看清机制如何选出那句温暖回复 | 定位本轮哪环出问题 |
| **批量总览** | 45条验收run聚合：情境准确率、落库计数、boundary拦截、缺陷闭环 | 方法可靠性证据 | 回归基线 |
| **框架对标** | C-SSRS三级 / EPITOME三维（情感vs认知共情分组，标注IP可靠性limitation）/ CASEL情境→维度映射 | 维度有权威背书、可引用 | 维度定义即文档 |

> **为什么升级成「研究工具」而非「评委视角」**：身份从「学生端B面」改为「研究/分析台」后，矛盾消失且能展示的内容更多（批量统计、框架对标在「B面」框架下放不进去）。它同时服务「向评委讲产品」与「开发者看状态」，是个真有日常用途的东西，因此更可信。

### 3.2 视觉：刻意与学生端不同

- editorial / 临床研究风：纸感暖底（`#f4f0e8`）、衬线显示字、信息密度高。
- **关键细节**：`boundary_flag=true` 的候选显式标「出局」、`weighted_total` 划除、不参与择优展示——直观呈现「安全/真实性优先于共情分」。

> **为什么反差是刻意的**：产品的温暖 vs 工具的严谨，这个对照本身是演示叙事——评委先以学生身份体验温暖，再走进分析台看清机制，体会到「刚才那句温暖的话是这套严谨机制选出来的」。

## 4. 重新开发的依据与边界

凡重新开发，以下为**不可协商项**（其余如布局细节、动效时长可在保持设计哲学下调整）：

1. **配色铁律**（§1.2）：暖白主、sage 仅点睛、绝不大面积绿填充。
2. **工艺规则**（§1.3）：无纯黑白、带色阴影、圆形语言。
3. **AI 回复层级**（§1.4）：大字独立成段，重于用户消息。
4. **学生端安全铁律**（§2.4）：三字段、无机制入口、不暴露错误。
5. **危机转介**（§2.3）：锁输入、含可信成年人、号码硬编码。
6. **两端物理分离**（§0）：不共享组件，只共享类型与 API 层。

可自由发挥但需说明理由的：空状态形式、起手式文案、emoji 取舍（若用，颜色须落在 sage/mist/clay 体系内，勿用体系外的紫等）、侧栏交互细节。

> **给重新开发者的一句话**：本产品的难点不是「做一个聊天框」，而是「让一个正在烦躁的初中生进来能松一口气，同时让评委相信背后有一套边界清晰、对孩子安全的严谨系统」。每个设计决策都应能回答「这如何服务这两个目标之一」。
